## Padronização de Código

### Renderizador JSON Padrão
Respostas de API seguem formato padronizado:
```json
// Sucesso (2xx)
{
  "success": true,
  "data": {...},
  "message": "Operação realizada com sucesso."
}

// Erro (4xx/5xx)
{
  "success": false,
  "errors": {...},
  "message": "Erro ao processar solicitação."
}
```

### UUID como Chave Primária
Todos os modelos herdam de `UUIDModel`:
-IDs não sequenciais (não expõe estrutura interna)
- Suporte a merged databases
- IDs universalmente únicos

## Validação de Modelos

### Choices Definidos
Todos os campos com conjunto fechado de valores:
- Status de perfis: pending, approved, rejected, suspended
- Tipos de usuário: PF, PJ, DRIVER, CARRIER, ADMIN, etc.
- Categorias de carga: EQUIPMENTS, DRY_GRAINS
- Tipos de veículos: 14 tipos diferentes
- Tipos de carroceria: 7 tipos
- Status de fretes: 13 estados no ciclo de vida

### Campo `created_at` / `updated_at`
Todos os modelos UUIDModel possuem timestamps automáticos

## Integridade de Dados

### Relacionamentos com ON DELETE
- `CASCADE` — Entidade dependente é excluída
- `SET_NULL` — Campo fica NULL (com null=True)
- Evitado `SET_DEFAULT` sem default definido

### Índices
- `unique=True` em campos críticos (email, CNPJ, placa, RENAVAM)
- `ordering` definido para consultas otimizadas

### Constraints de Negócio
- `max_length` para strings
- `max_digits/decimal_places` para decimais
- `PositiveIntegerField` para contadores
